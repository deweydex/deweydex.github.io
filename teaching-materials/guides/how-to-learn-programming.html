How to Learn Programming

A practical guide to learning programming in the mid-2020s

Introduction

This guide will hopefully be a little bit different from most guides out there on how to learn programming in the mid 2020s. I will indeed give some concrete advice about what one can use and steps to get started, but the Resources and Recommendations Page contains more context specific suggestions for specific subjects and use cases.

What Do You Mean By Programming?

In my experience there are at least five types of programming: they are certainly all related, but the task of "learning programming" usually involves starting to learn one of these broader types:

  1. Programming Contests or Puzzles
  2. Scientific or Financial Programming
  3. Programming Applications (Back-end, mid-layer, native)
  4. Graphics, Front-End and Web Programming
  5. Low-Level Programming

Note that some areas require practices and ideas and languages from more than one of these broad types. In particular, learning how to create computer games likely involves learning several ways of thinking about programming, especially graphics heavy games or games involving complex mechanics.

My Philosophy

There are many philosophies and opinions around what one ought to learn first. I don't really ascribe to a prescriptive sense of "should" here: learn whatever gets you excited and interested so that you can continue!

Aside: if you are learning how to program in order to accomplish something specific, like create a game or an app, diving into one of the other areas of programming might be a nice smooth introduction. In my courses we focus on learning how to learn programming and the connected subjects like math and computer science.

A Coding Process for Bigger Challenges

When confronted with a programming challenge, especially one which appears to be pretty big and where you aren't sure where to begin, there are a few basic rules of thumb which can do wonders for your confidence and get you on a path toward both understanding the scope of the problems you need to solve and what order you might best approach those problems.

Planning Phase

  1. Write the introduction or assignment brief in your own words: This will help you to highlight the parts you find important and also give you a chance to figure out what you don't understand. If the brief is very vague or big like "build a website for yourself" or "create a map for a game and the functions behind it," start by writing out (in words) the smallest simplest thing you can think of that might get the job done.
  2. Think through variables and data types: As you think through a problem, note down what variables you might need and what type of variables they might be. If you are not sure of what datatype to use, think through what you need the variables to do, what they need to store, how they will be manipulated, and then have a look at some documentation about various data structures that might solve that problem. To start, pick the simplest, even if it doesn't meet all of your needs—better to start with something than nothing!
  3. List the behaviors: Write out a list of the behaviours you would like your program to accomplish. See if you can break down each bigger behaviour like "tell the user where they are located" into as many substeps as you can. For example: provide a prompt to the user, receive user input, process that input amongst a few options (including "where am I?"), and provide an answer to the user based on a variable that we have stored earlier (if you haven't stored a variable, might want to put that at the top of your list of steps).
  4. Think through conditions and functions: Think through the conditions (if/else statements) you need to check—what variables do you need to check and for what property or amount? If you are able to see how those behaviours might map to functions, think through what you need each function to take as input (what types are those variables, if any?) and what type you need them to output. Feel free to have these if-then statements and function definitions kind of hanging loose in your editor with "TODO" comments showing what needs to happen before and after.

Note

All of the above can be done in any text document, or on paper! If you would like to try just jumping into code that's fine—some folks are planners, and others write lots of comments in their code that go over these things.

When you get to the actual coding, don't be afraid to use the simplest text editor you can! Especially in the beginning, a fully fledged IDE can seem a bit overwhelming. For my courses I recommend using online "lite" IDEs first, whether Jupyter Notebooks, online C# interpreters, Replit, and more.

Coding Phase

  1. Start defining variables and functions with names that make sense to you. A good variable name is one that is descriptive and often might be longer than we would like. The worst thing is an abbreviated name that you can't recognize later!
  2. Comments are your friends! Even multi-line comments. Describe your thoughts and intentions as you code—this will help you solve problems before they happen and figure out changes you might need to make later on.
  3. Print statements are also your friends! If you need to check the state of a variable before an if statement or during a loop, go ahead and print it out with a little string that says where that print-out happened! Digital ink is free—don't be afraid to have lots of little statements saying what is happening throughout your code.
  4. Not knowing what is next is normal. Not knowing where to begin is normal. Starting is more important than being correct. Writing a plan will help you find your footing but also allow you to ask for help more effectively. "I don't get it" is not a valid sentence if you are asking for help, but "I am stuck on this part of the plan because I don't understand how to do X" shows the person you are asking that you are trying to point them in the right direction.
  5. Everyone uses documentation. Learning how documentation works is part of learning how to code and how to solve coding challenges. Not remembering something is to be expected; not knowing how to find something just isn't necessary anymore. Using documentation to figure out what methods and functions exist and how to use them is what the beginning, middle, and end of learning to code looks like—just the documentation and the challenges you try to solve get more involved and complex.
  6. Complete the minimum required first. If there is a brief, complete the minimum required for the brief first, then start expanding and creating more elaborate stuff. It can be too easy to get lost in the part of the project you enjoy and forget the rest of an assignment. In the case of coding, the minimum requirements are usually enough—all that is above that is to be encouraged, but is then about your learning, not fulfilling an assignment.
